# VirtualizedList family — shared-layer placement note

The VirtualizedList family (`VirtualizedList`, `FlatList`, `VirtualizedSectionList`,
`SectionList`) is brought to **full feature parity** across the React and Vue adapters by
extracting all framework-agnostic logic into `core/components/src/state/`. This records the
one structural deviation from the usual 3-layer split and confirms coverage.

## Why lists have STATE but no `view/render-*.ts`

The standard split (`<components_split_logic_view_lifecycle>`) is: pure reducer in
`state/`, pure `render*(viewState) -> Descriptor` in `view/`, lifecycle in the adapter.

Lists break the `view/` layer on purpose: a list cell's **content is the framework's own
children** — `renderItem({ item, … })` returns a `ReactNode` in React and a `VNode` in Vue.
A `Descriptor` child is only `Descriptor | string` (see `descriptor.ts`), so a framework
element cannot be embedded in a Descriptor tree, and `descriptorToVue` / `descriptorToReact`
cannot bridge one. There is therefore no single prop-driven render function that produces the
whole list.

So the shared layer for lists is the **STATE/logic**, not a Descriptor render fn:

- `state/virtualized-list.ts` — defaults/sentinels, nativeEvent readers, offset table
  (`buildOffsets`), window compute (`computeWindow`) + batch throttle (`throttleWindow`),
  viewability classification + set diff + `minimumViewTime` fold, edge-reached geometry,
  `offsetForIndex`, and `buildListPlan` (spacer extents + in-window cell keys + sticky child
  positions). Plus the shared data + imperative-handle types.
- `state/flat-list.ts` — `chunkIntoRows`, row-token viewability expansion, row-separator
  item unwrap.
- `state/section-list.ts` — `flattenSections`, entry key/unwrap, `scrollLocationToFlatIndex`,
  the section handle type.

`buildListPlan` carries the shared half of the render (every windowing decision); the
adapter walks the plan and creates host elements (`createElement` / `h`) plus the user's
`renderItem` content. That is the irreducible per-framework part — element creation and the
user render callback — and nothing more. A windowing/viewability/edge bug is fixed once.

## Coverage — FULL (no deferral)

All four components, both adapters, share the core above:

| Capability | Source of truth |
|---|---|
| windowing, spacers, batch fill | `computeWindow` / `throttleWindow` / `buildListPlan` |
| `getItemLayout` + estimated-height fallback | `buildOffsets` / `averageMeasuredLength` |
| viewability + `onViewableItemsChanged` diff + `minimumViewTime` + `waitForInteraction` | `computeViewableSet` / `diffViewable` / `maxMinimumViewTime` |
| `onEndReached` / `onStartReached` | `computeEndReached` / `computeStartReached` |
| imperative handles (`scrollToIndex/Item/Offset/End`, `flash`, `getScroll*`, `recordInteraction`, `scrollToLocation`) | `IVirtualizedListHandle` / `IVirtualizedSectionListHandle` + `offsetForIndex` / `scrollLocationToFlatIndex` |
| sticky headers, inverted, MVCP, RefreshControl, separators | plan sticky positions + adapter wiring on the shared ScrollView |
| `numColumns`, multi-column viewability/separators | `chunkIntoRows` / `expandRowViewability` |
| section flatten, section separators, sticky section headers | `flattenSections` |

The framework-specific prop interfaces (`IVirtualizedListProps`, …) stay per-adapter because
they reference `ReactNode` / `VNode` and the framework component types; the LOGIC and HANDLE
surface they wrap are shared. Parity is structural.
