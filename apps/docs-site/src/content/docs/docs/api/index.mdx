SymbioteJS has one native engine, but each adapter exposes that engine through its
framework's API conventions. The reference is organized around that split.

## Framework APIs

- [React API](/symbiotejs/docs/api/react/) — callback props, React children, React refs,
  and React Native-like imports from `@symbiotejs/react`.
- [Vue API](/symbiotejs/docs/api/vue/) — typed emits, slots, Vue refs, kebab-case template
  attributes, and imports from `@symbiotejs/vue`.
- [Angular API](/symbiotejs/docs/api/angular/) — real `@Output()` `EventEmitter`s,
  `<ng-content>` projection, standalone components, and imports from
  `@symbiotejs/angular`.

## Shared APIs

- [Components matrix](/symbiotejs/docs/api/components/) — what components exist and how
  React, Vue, and Angular differ for props, events, slots, and handles.
- [Core API](/symbiotejs/docs/api/core/) — `@symbiotejs/engine` and `@symbiotejs/components`,
  the shared layers that keep adapters thin.

## How to read component docs

Each component API should answer four questions:

1. What props are shared across frameworks?
2. Which events are React callbacks vs Vue emits?
3. Does the component take children, slots, or render callbacks?
4. Does it expose an imperative handle, and how does each framework access it?

This is why the docs avoid pretending every framework has identical props. The
native behavior is shared; the public contract is idiomatic per framework.
